## How to use / how it fits

Nothing new runs on its own — these files are the retrieval corpus for the analyzer sprint (10). `SeoPlaybookIndex.load()` already `readdir`s every `skills/bober.seo-*/SKILL.md`, so these three appear in the flat signature list and in `SeoPlaybookRetriever.retrieve({ workflow })` results the moment they exist on disk. A `retrieve({ workflow: "technical-audit" })` call now returns these 8 signatures ranked ahead of the generic floor (appended, deduped by `playbookId`). Retrieval is workflow-tagged: every block declares its `Workflows:` line matching the skill's workflow, which the new `skills-content.test.ts` asserts.

## Notes for maintainers

- **Citation discipline is the load-bearing invariant here, and it is enforced by the evaluator, not the parser.** The parser only checks a `PrimarySourceUrl` is present and non-empty — it cannot tell a real slug from a plausible hallucinated one (this is exactly what burned Sprint 2's iteration 1). Every one of the 9 unique URLs across these three files was copied verbatim from the briefing's verified map and re-`curl`ed to 200 before commit. If you edit a citation, `curl -sI` it for a `200` first.
- **`policyClass` discipline: auto-safe vs human-approve.** The split is not cosmetic — auto-safe blocks are read-only audits that an analyzer may act on directly; the two `human-approve` blocks (`authority-consolidation-merge`, `mass-schema-generation-approval`) gate the only tactics that mutate a site (redirects/migrations, bulk markup deployment). No `never-encode` tactic appears in these files, and no site-mutating tactic is tagged `auto-safe`. Do not downgrade a `human-approve` block without the same human-approval reasoning in the invariant.
- **Evidence grades:** none of the 21 blocks is graded `verified` — they are `single-source` (15, mostly the single Content Warehouse leak analyses at ipullrank.com / hobo-web.co.uk / sparktoro.com) or `primary-unverified` (6, Google's own developer docs). The leak-derived signals are inherently single-source; treat them as such.
- **These skill files and `SeoPlaybookParser` are one executable spec.** The block format lives in `skills/bober.seo-generic/SKILL.md` ("Signature Block Format"); a format change must be mirrored in the parser and vice versa. Each of these three files points back to it in its header.
- **The `playbook-index.test.ts` relaxation is legitimate, not a weakening.** The old assertion required *every* discovered signature's `skillRef` to contain `bober.seo-generic` — true only while the generic floor was the sole skill. It was relaxed to `bober.seo-` (every discovered skill is still a `bober.seo-*` skill) plus a new positive assertion that the generic floor is *among* the discovered signatures. The `>=10` floor is unchanged. This was required for the index test to survive the arrival of the three non-generic skills.
