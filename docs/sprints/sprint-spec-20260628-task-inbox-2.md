# Task listing + lifecycle transitions (start / done / drop)

**Contract:** sprint-spec-20260628-task-inbox-2  ┬Ě  **Spec:** spec-20260628-task-inbox  ┬Ě  **Completed:** 2026-06-29

## What this sprint added

Sprint 2 makes the **task-inbox** usable past capture: a user can now **see** their active tasks
and **move them through a lifecycle**. It adds a single PURE `transitionFinding` helper
(`src/hub/finding-store.ts`) that supersedes a task's active Finding with a new-status copy through
the reconcile **UPDATE** path ÔÇö so the prior status survives as queryable bitemporal history ÔÇö plus
the `bober task list` command and the `bober task start|done|drop <id>` transition subcommands
(`src/cli/commands/task.ts`). The default list hides terminal (`done`/`dropped`) tasks; `--all` and
`--status <s>` widen it. **No** row is ever deleted ÔÇö `drop` transitions to `status='dropped'` via
supersede.

## Public surface

- **`bober task list [--all] [--status <status>]`** (`src/cli/commands/task.ts:203`,
  `registerTaskCommand`) ÔÇö print a table of tasks. **Default** shows only `open` + `in-progress`
  Findings; `--all` shows every status (including `done`/`dropped`); `--status <s>` shows only that
  status (takes precedence over `--all`). Columns: `ID  STATUS  DOMAIN  TITLE` (title truncated at 36
  chars). Prints `No tasks found.` when the filtered set is empty.
- **`bober task start <id>`** (`src/cli/commands/task.ts:234`) ÔÇö transition the task to
  `status='in-progress'`.
- **`bober task done <id>`** (`src/cli/commands/task.ts:234`) ÔÇö transition the task to
  `status='done'`.
- **`bober task drop <id>`** (`src/cli/commands/task.ts:234`) ÔÇö abandon the task by transitioning
  it to `status='dropped'` **via supersede ÔÇö never `DELETE`**.
  (start/done/drop are generated by one loop, so they share a registration site.)
  On an unknown id every transition prints a `chalk.yellow` "no task found" message and sets
  `process.exitCode = 1` **without throwing**.
- `transitionFinding(store, id, newStatus, { now, mutate? })` (`src/hub/finding-store.ts:62`) ÔÇö
  the PURE core. Reads the active Finding for `id` via `readFindings`; returns **`null`** if none is
  active. Otherwise builds `{ ...current, ...mutate, status: newStatus }` and writes it back through
  `writeFinding` Ôćĺ `writeFact`. Because `subject=id` and `predicate='finding'` are unchanged but the
  serialized `value` differs, `reconcileFact` takes the **UPDATE** branch (supersede old row + insert
  new active row) ÔÇö preserving the prior status as a `t_invalidated` history row. Returns the new
  `Finding`. **Never** reads the clock ÔÇö `now` is injected from the CLI boundary.
- `runTaskList(store, { all?, status? })` (`src/cli/commands/task.ts:97`) ÔÇö synchronous, read-only
  DI core for `task list`; needs no `now`. Filters per the rules above and writes the table to
  stdout. Catches all errors to stderr with `exitCode=1`; never throws.
- `runTaskTransition(store, id, newStatus, now)` (`src/cli/commands/task.ts:138`) ÔÇö DI core for
  start/done/drop. Delegates to `transitionFinding`, maps a `null` result to the not-found path
  (`exitCode=1`), prints `Task <id> Ôćĺ <status>` on success. Never throws.
- `ACTIVE_STATUSES` (`src/cli/commands/task.ts:89`) ÔÇö module-private `['open', 'in-progress']`; the
  default-list filter.

## How to use / how it fits

```bash
$ bober task add "renew passport"
Captured task 1f3c9a0b2e4d6f80

$ bober task list
ID                 STATUS       DOMAIN       TITLE
--------------------------------------------------------------------------------
1f3c9a0b2e4d6f80   open         inbox        renew passport

$ bober task start 1f3c9a0b2e4d6f80
Task 1f3c9a0b2e4d6f80 Ôćĺ in-progress

$ bober task done 1f3c9a0b2e4d6f80
Task 1f3c9a0b2e4d6f80 Ôćĺ done

$ bober task list                # done task is now hidden
No tasks found.

$ bober task list --all          # ÔÇŽbut still in the store as history
ID                 STATUS       DOMAIN       TITLE
--------------------------------------------------------------------------------
1f3c9a0b2e4d6f80   done         inbox        renew passport
```

The transition commands consume the same hub pool that `bober task add` (Sprint 1) produces and that
`bober hub list` / `priority` / `decide` / `chat hub` read. Each handler opens the active team's
namespace `FactStore`, stamps `now = new Date().toISOString()` at the **boundary only**, delegates to
the pure `runTaskTransition` Ôćĺ `transitionFinding` Ôćĺ `writeFinding` chain, and closes the store in a
`finally`. `task list` is read-only and needs no `now`.

## Notes for maintainers

- **Lifecycle is supersede, not delete.** `done`/`drop` go through the reconcile UPDATE branch, so a
  terminal task is still an **active row** carrying `status` `done`/`dropped`, and the earlier status
  lives on as a superseded (`t_invalidated` set) row. The list filters by the **`status` field**, not
  by invalidation. Do not "optimize" `drop` into an `invalidateFact`/`DELETE` ÔÇö that would erase the
  history the contract requires (verified by `sc-2-2`).
- **`now` stays out of the hub layer.** `transitionFinding` is PURE and takes `now` as a parameter,
  matching `writeFinding`/`captureTask` from Sprint 1. Keep clock reads at the CLI boundary.
- **`transitionFinding` reuses `readFindings`, which is strict.** It will throw on a malformed stored
  row (`FindingSchema.parse`). That is intentional for own-store reads; the cross-repo listing path
  (`FactStoreFindingSource`) is the lenient one. See the Sprint 1 record for the contrast.
- **`--status` beats `--all`.** When both are passed, `--status` wins (it is checked first); there is
  no validation that the status string is a real `Finding["status"]` ÔÇö an unknown status simply
  yields `No tasks found.`
- **Iteration-2 lint fix.** The original implementation passed all 5 criteria on iteration 1; the
  only failure was a lint hard-gate (unused `task2` binding in `task.test.ts`). Commit `26f45db`
  added `expect(output).toContain(task2.id)`, clearing the error and strengthening the `sc-2-3`
  assertion. No behavior changed.
- **Snooze + ingest are still later sprints.** `snooze` wake semantics (Sprint 3) and domain-finding
  ingest (Sprint 4) are explicitly out of scope here.
